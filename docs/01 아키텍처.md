# 01 아키텍처

## 전체 데이터 흐름

```
[사용자 입력]
  주제 · 검색 키워드 · 레퍼런스 URL · 첨부 파일 · (자동 탐색 결과)
        │
        ▼
[레퍼런스 수집]  webfetch.py + filerefs.py + discovery.py
  → 모든 LLM에게 동일한 원문 텍스트 제공 (reference_texts dict)
        │
        ▼
[① 조사]  pipeline.run_research()
  각 프로바이더 병렬 실행 (ThreadPoolExecutor)
  키워드 있음 → 각자 자체 웹 검색 도구 사용
        │
        ▼
[② 토론]  pipeline.run_discussion()  (0~3라운드)
  서로의 결과를 익명(연구원 A/B/C)으로 교차 검토
  라운드마다 "수정된 최종 입장"으로 갱신
        │
        ▼
[③ 종합]  pipeline.run_synthesis()
  진행자(moderator) LLM이 보고서 JSON 생성 (REPORT_SCHEMA)
  분량 계획(_length_plan)에 따라 섹션/표 개수·서술량 지시
        │
        ▼
[④ 출력]
  로컬: reports/ (python-pptx · python-docx · openpyxl)
  외부: ppt_engines/ (Gamma · Canva · Google Slides)
```

## 모듈 맵

```
app.py                        Streamlit UI (입력/진행/결과/다운로드)
core/
  config.py                   ProviderSpec 정의, API 키 감지, 기본 페르소나
  pipeline.py                 ★ 핵심 — 조사/토론/종합 오케스트레이션, 스키마, 분량계획
  discovery.py                레퍼런스 자동 탐색 (+ 프로바이더 폴백)
  webfetch.py                 URL 본문 추출 (requests + BeautifulSoup)
  filerefs.py                 첨부 파일 본문 추출 (pypdf/docx/pptx/openpyxl)
  providers/
    base.py                   BaseProvider 인터페이스 + parse_json_loose
    anthropic_provider.py     Claude (웹검색 서버도구 + 구조화 출력 + pause_turn)
    openai_provider.py        GPT (Responses API + web_search, Chat 폴백)
    gemini_provider.py        Gemini (Search 그라운딩, thinking 토큰 방어)
  reports/
    pptx_builder.py           PPT — 표준/컴팩트/원페이저 3종 레이아웃
    docx_builder.py           Word — 한글 폰트(eastAsia) 처리 포함
    xlsx_builder.py           Excel — 요약/본문/데이터/출처 시트
  ppt_engines/
    base.py                   PPTEngine 인터페이스 + report_to_markdown
```

## 핵심 설계 결정과 이유

1. **레퍼런스는 로컬에서 추출해 전 LLM에 동일 제공**
   프로바이더마다 URL fetch 능력이 달라, 프로그램이 직접 본문을 추출해
   같은 원문을 주면 조사 조건이 공평해지고 토론 품질이 올라간다.

2. **토론은 익명(연구원 A/B/C)으로 진행**
   모델명을 노출하면 브랜드 편향(예: GPT가 Claude 결과를 과대평가)이
   생길 수 있어 익명화. UI에는 실명 표시.

3. **병렬 실행은 ThreadPoolExecutor**
   3사 SDK 모두 동기 클라이언트 사용. Streamlit 환경에서 asyncio보다
   단순·안정적. UI 업데이트는 as_completed 루프(메인 스레드)에서 처리.

4. **프로바이더 실패는 격리**
   개별 조사/토론 실패는 오류로 기록하고 파이프라인은 계속 진행.
   진행자는 성공한 프로바이더 중에서만 선택. → [[03 LLM 프로바이더]]

5. **보고서는 구조화 JSON을 거쳐 생성**
   LLM이 직접 파일을 만들지 않고 REPORT_SCHEMA JSON을 생성 →
   빌더가 결정적으로 렌더링. 형식 3종 + 외부 엔진이 같은 JSON을 공유.
   → [[02 파이프라인 설계]], [[05 보고서 생성]]
