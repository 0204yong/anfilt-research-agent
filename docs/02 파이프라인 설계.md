# 02 파이프라인 설계

파일: `core/pipeline.py`

## 데이터 모델

| 클래스 | 역할 |
|---|---|
| `ResearchBrief` | topic, keywords, reference_urls, reference_texts{url:본문}, instructions, persona |
| `AgentFinding` | 조사 결과 — provider_key/label, model, text, error |
| `DiscussionTurn` | 토론 발언 — round_no, provider, text, error |
| `PipelineResult` | findings, discussion, scorecard(채점표 JSON), report(JSON), moderator_label, anon_map{"연구원 A": 실제 LLM 라벨} |

## ① 조사 `run_research()`

- 전 프로바이더 병렬 (`ThreadPoolExecutor`)
- **키워드가 있으면** `web_search=True` — 각 LLM의 자체 검색 도구 사용
- 프롬프트 구조: 브리프 블록 + 출력 형식 지시
  (핵심 요약 / 주요 발견 / 세부 분석 / 데이터·수치 / 출처 목록)
- 실패는 `AgentFinding.error`에 기록하고 계속 진행

## ② 토론 `run_discussion()`

- 성공한 프로바이더가 **2개 이상**일 때만 진행 (아니면 빈 리스트)
- 라운드마다: 각자 [자신의 최신 입장 + 동료들의 최신 입장(익명)]을 받아
  1) 동의/이견 2) 보완 3) **수정된 최종 입장** 을 작성
- `current` dict가 라운드마다 갱신 → 다음 라운드 입력
- 프롬프트에 "근거 없는 양보 금지" 명시 (동조 편향 방지)

## ③ 채점 `run_scoring()` (2026-07-04 추가)

- 종합 **직전에** 진행자 LLM이 연구원별 최종 입장을 공식 채점표로 평가
- 채점 기준: `SCORING_CRITERIA` 9종 중 UI 선택 (기본 `DEFAULT_CRITERIA_KEYS` 6종)

### 채점 기준 9종 (`SCORING_CRITERIA`)

| # | key | 기준 | 평가 내용 | 기본 |
|---|---|---|---|---|
| 1 | `accuracy` | 정확성 | 사실관계가 정확한가. 토론에서 반박당하고도 수정하지 않은 오류가 없는가 | ✅ |
| 2 | `evidence` | 근거·출처 신뢰도 | 공식 기관·1차 출처를 인용했는가. 핵심 수치마다 출처가 붙어 있는가 | ✅ |
| 3 | `completeness` | 완결성 | 브리프가 요구한 범위(요약·발견사항·세부 분석·데이터·출처)를 빠짐없이 다뤘는가 | ✅ |
| 4 | `recency` | 최신성 | 최신 데이터·규정·동향을 반영했는가. 낡은 수치를 최신인 것처럼 쓰지 않았는가 | ✅ |
| 5 | `quant` | 정량성 | 정량 데이터(수치·연도·금액·비율)가 구체적이고 표로 정리 가능한 수준인가 | ✅ |
| 6 | `logic` | 논리 일관성 | 주장과 근거가 정합하는가. 문서 안에서 서로 모순되는 내용이 없는가 | ✅ |
| 7 | `actionability` | 실행가능성 | 제언이 고객사가 바로 실행에 옮길 수 있을 만큼 구체적인가 | 선택 |
| 8 | `balance` | 균형·객관성 | 반론·한계·불확실성을 숨기지 않고 명시했는가. 한쪽 시각에 치우치지 않았는가 | 선택 |
| 9 | `responsiveness` | 토론 수용성 | 토론에서 받은 지적을 반영해 개선했는가, 또는 근거를 들어 제대로 방어했는가 | 선택 |

- 채점 원칙(프롬프트에 명시): 모두에게 후한 점수 금지·기준별 차등, 출처 없는
  수치·미확인 주장·토론에서 반박당하고 방치한 오류는 명시적 감점 요인,
  총점 동률 근접 시 정확성·근거 점수 우선
- 기준 추가/수정 방법: `core/pipeline.py`의 `SCORING_CRITERIA`에 dict 한 줄
  (`key`/`label`/`desc`) 추가하면 UI multiselect에 자동 노출.
  기본 선택에 넣으려면 `DEFAULT_CRITERIA_KEYS`에 key 추가
- 출력: `evaluations[]`(연구원별 기준당 score 1~10 + comment, strengths/weaknesses)
  + `best`(선정) + `rationale`(선정 이유 3~5문장)
- **점수는 코드에서 1~10 클램프, 총점(total)도 코드로 재계산** — LLM 산술 불신
- 스키마에 min/max 수치 제약을 넣지 않음(프로바이더별 지원 상이) —
  프롬프트 지시 + 클램프로 보장. researcher/criterion은 enum으로 제한
- 조건: 성공한 연구원 ≥ 2 && 기준 ≥ 1개. 채점 실패 시 **채점표 없이 종합 계속**
  (try/except — 채점이 보고서 생성을 막으면 안 됨)
- 채점표는 ④종합 프롬프트에 "사전 채점 결과" 블록으로 주입되고,
  UI의 **🏅 채점표 탭**에 기준×연구원 매트릭스 + 기준별 사유로 공개됨

## ④ 종합 `run_synthesis()`

- 진행자 우선순위: `pick_moderator()` — Claude(구조화 출력 지원) → GPT → Gemini
- 단, 조사에 성공한 프로바이더 중에서만 선택
- 종합 방식 2가지 (UI 라디오):
  - `synthesize` — 교차 검증 통합 (공통 확인 내용 우선, 상충 시 근거 강한 쪽 —
    채점표가 있으면 정확성·근거 점수가 높은 연구원 우선 검토)
  - `best` — 채점표의 베스트 연구원 결과 중심 + 검증된 보완만 반영,
    채점표가 지적한 약점은 다른 연구원의 검증된 내용으로 보완
- 결과는 `REPORT_SCHEMA` JSON:
  - Claude: `output_config.format` 구조화 출력 (스키마 보장)
  - GPT/Gemini: 프롬프트 지시 + `parse_json_loose()` (코드펜스/잡음 방어 파싱)

## REPORT_SCHEMA (보고서 JSON)

```json
{
  "title": "...", "subtitle": "...",
  "executive_summary": "...",
  "key_findings": ["..."],
  "sections": [{"heading", "content", "bullets": []}],
  "data_tables": [{"title", "headers": [], "rows": [[]]}],
  "recommendations": ["..."],
  "sources": [{"title", "url"}]
}
```

- 모든 객체에 `additionalProperties: false` + 전 필드 required
  (Claude 구조화 출력 요구사항)
- 표의 셀은 전부 문자열 (렌더링 단순화)

## 분량 계획 `_length_plan(target_pages)`

목표 PPT 장수(1~20)를 받아 구성 요소 개수를 역산 — [[05 보고서 생성]] 참고.

| 구간 | PPT 구성 | 산식 |
|---|---|---|
| 1장 | 원페이저 | sections=0, tables=0, 요약 3~4문장 |
| 2~7장 | [표지+요약] + 섹션/표 + [발견+제언] | sections + tables = target − 2 |
| 8~20장 | 표준 5장 고정 + 섹션/표 | sections + tables = target − 5 |

서술 깊이(섹션당 글자 수, bullets 수)도 구간별로 지시 →
Word 분량도 비례해서 조절된다. 진행자 프롬프트에 "반드시 준수"로 삽입.

## 실행 진입점

```python
run_pipeline(providers, brief, rounds=1, mode="synthesize",
             target_pages=12, criteria=None, on_update=콜백) -> PipelineResult
```

`criteria`: `SCORING_CRITERIA` 항목 리스트. None=기본 6종, 빈 리스트=채점 생략.

`on_update(str)`는 UI 진행 표시용 — Streamlit 밖(배치/CLI)에서도 사용 가능하게
파이프라인은 Streamlit에 의존하지 않는다.
