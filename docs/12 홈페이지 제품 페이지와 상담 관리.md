## 페이지 구조 (2026-07-04부터)

```
소스 (Company_Homepage 저장소)          배포 URL (Netlify 빌드)
─────────────────────────────          ──────────────────────
concept-c.html   (랜딩)            →   /            (dist/index.html)
product.html     (제품)            →   /product/    (dist/product/index.html)
admin.html       (관리자·비공개링크) →   /admin/      (dist/admin/index.html)
assets/site.css  (공용 스타일)      →   /assets/site.css
```

- 공용 CSS는 concept-c.html 인라인 `<style>`에서 **verbatim 추출**한 `assets/site.css`.
  세 페이지가 `<link href="/assets/site.css">`(절대경로)로 공유 — 스타일 수정은 이 파일 한 곳에서.
- 랜딩 나브·푸터의 "제품"은 `/product/` 링크. 제품 페이지의 나머지 메뉴는 `/#about` 등
  랜딩 해시로 되돌아간다.
- Research Agent 실행 링크(`APP_URL` 스크립트)는 product.html 끝으로 이동
  (비우면 "도입 문의"→/#contact, 채우면 앱 링크 — 기존 철학 유지).
- admin.html은 어디에도 링크하지 않음 + `noindex, nofollow`. 주소를 아는 사람만 접근하되
  실제 보안은 아래 로그인+RLS가 담당.

## 상담 신청 데이터 흐름

```
방문자 상담폼 제출 ─┬─ Web3Forms → 대표 메일 (기존 유지)
                   └─ Supabase leads INSERT (익명, RLS로 쓰기만 허용)

관리자(/admin/) ── Supabase Auth 로그인(이메일/비번)
                └─ leads SELECT/UPDATE — RLS가 "관리자 이메일 JWT"일 때만 허용
                   (회신 완료 토글 replied/replied_at, 메모 admin_note)
```

## 관리자 화면 (admin.html)

- 로그인: `supabase.auth.signInWithPassword` — 세션은 localStorage 유지(새로고침 시 자동 복원)
- 목록: 신청일시·회사명·담당자·연락처·내용, 최신순. 필터(전체/대기/완료) + 건수 요약
- 관리: "회신 완료로 표시" 토글(replied, replied_at 기록), 메모(admin_note, 500자)
- 방어: 모든 신청자 입력은 esc() 이스케이프(XSS), RLS 거부 시 update 0행을
  `.select()` 결과로 감지해 실패 표시(성공 위장 금지), DELETE 정책 없음(감사 추적 보존)

## Supabase 마이그레이션 (supabase-admin-setup.sql)

1. `leads`에 `replied boolean default false` / `replied_at timestamptz` /
   `admin_note text default ''`(≤500자) 추가
2. 정책: `leads admin select` / `leads admin update` — `authenticated` 역할 중
   `auth.jwt()->>'email' = '0204yongko@gmail.com'` 만 허용 (이메일 변경 시 SQL 수정 후 재실행)
3. 익명 INSERT 정책 강화: `replied=false and replied_at is null and admin_note=''` 강제
   → 신청자가 회신 상태 위조 불가 (폼은 해당 컬럼을 안 보내므로 기본값으로 통과)

## 사용자(대표) 1회 설정 — 이것 전까지 /admin/ 로그인 불가

1. Supabase SQL Editor에서 `supabase-admin-setup.sql` 실행
2. Authentication → Users → **Add user**: SQL의 관리자 이메일과 동일 + 비밀번호 +
   Auto Confirm User 체크 (계정 생성·비밀번호는 규정상 에이전트 대행 불가)
3. (권장) Authentication → Sign In / Up 에서 공개 회원가입 끄기

상세 절차는 홈페이지 저장소 `SETUP.md`의 4-1단계에 동일하게 기록됨.

## 이메일 제거 내역

- 상담 섹션 "직접 연락 — mailto" 줄 삭제, 푸터 문의란 mailto 삭제(→ 무료 상담/게시판 링크로 대체)
- 폼 전송 실패 안내문에서 이메일 제거(→ "문의 게시판에 남겨 주세요")
- `CONFIG.CONTACT_EMAIL` 항목 자체 삭제, 주석 속 이메일도 제거
- Web3Forms 발송 대상 이메일은 Web3Forms 계정 설정(서버측)이라 화면 비노출 — 메일 수신은 계속됨
