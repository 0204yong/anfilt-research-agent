# 13 지식 볼트와 온톨로지

> 상태: **1~3단계 구현·라이브 검증 완료 (2026-07-11)** — 로컬 활성화됨.
> 4단계(지식 주입)부터 미착수. 배포 반영은 push 대기.
> 이 볼트는 풀 모드와 **라이트 모드(비용 절감형, 방향 확정 2026-07-11)** 의
> 공용 기반이다 — 라이트 모드는 볼트 검색을 1차 답변으로 쓴다 (→ [[08 작업 이력]] 16번).
> 조사 결과가 세션 종료와 함께 사라지는 문제를 풀고, 사용자별 ESG 온톨로지를
> Obsidian 볼트로 축적해 다음 조사가 그 위에서 더 깊고 정확해지게 한다.

## 요구사항 (사용자, 2026-07-06)

1. 리서치 결과물이 사라지지 않고 저장되어 나중에 다시 불러올 수 있을 것
2. 사용자마다 자신만의 온톨로지가 Obsidian 볼트 형태로 축적될 것
3. 유사한 주제를 다시 조사하면 축적된 지식 위에서 더 깊은 조사가 될 것

설계 과정: 코드 분석 4건 → 독립 설계안 3건(파일 우선 / Supabase 우선 /
지식그래프 우선) → 3관점 심사(사용자 적합성 / 엔지니어링 / 지식 품질)의
멀티에이전트 워크플로로 도출한 하이브리드안.

## 현재 상태 진단 (설계의 근거)

- 결과는 `st.session_state["result"]`에만 존재 — 새 조사 시작 시 pop되어 소멸
  (app.py:283-284). 보고서 3종은 BytesIO bytes만 반환, 디스크 저장 전무
- **`PipelineResult`에 `ResearchBrief`와 실행 파라미터(rounds/mode/criteria)가
  포함되지 않음** → 저장 훅은 run_pipeline 호출 지점(app.py:328 부근)에서
  둘을 함께 캡처해야 함
- 프로바이더 3사 모두 API 차원의 citations를 버림 — 출처는 `report.sources`의
  {title, url}이 전부. 수집 시각·해시 등 프로비넌스는 저장 시점에 새로 기록 필요
- Streamlit Cloud는 파일시스템 휘발성 → 영속화는 외부 저장소 필수
- Supabase(`unjhdoulorbmpepnkcpr`)는 현재 상담 관리(leads) 전용 — 재사용 가능
- 인증은 공유 비밀번호 1개뿐 → "사용자" 개념 없음 (다중 사용자는 후순위 과제)

## 확정 설계 — 이중 구조

역할이 다른 두 층으로 분리한다. 하나로 합치면 어느 쪽이든 무리가 온다.

### 1층 — 실행 아카이브 (기계용 원본, Supabase)

조사 실행 1건 = jsonb 1행 (`ra_runs` 테이블).
`brief`(주제·키워드·레퍼런스 원문) + `params`(rounds/mode/target_pages/
criteria_keys/providers/executed_at) + `result`(asdict(PipelineResult) 전체).

- 용도: 재열람(기존 5탭 UI로 복원), "최근 3개월 A사", "점수 8점 이상" 필터 질의
- 훅: 파이프라인 성공 직후 1회 + `_saved_run_id` 플래그(rerun 중복 저장 방지)
- 저장 로직은 `core/store.py`에 Streamlit 비의존으로 (→ [[02 파이프라인 설계]] 원칙)
- `schema_version` 필드 필수 (스키마 변경 후에도 과거 run 복원 가능하게)
- 레퍼런스별 `fetched_at`·`content_hash`·`is_error`('[' 접두사 관례 필터) 기록
- 백로그 ★★ "조사 이력 저장" 항목이 이것으로 해소됨

### 2층 — 지식 볼트 (사람용 온톨로지, Obsidian 마크다운이 원본)

**"노트 하나 = 엔티티 하나"** 원칙. 마크다운이 원본이고 사용자가 Obsidian에서
고치면 다음 조사에 그대로 반영된다(편집이 곧 온톨로지 큐레이션 — 별도 관리 UI 불필요).

```
지식볼트/
├── entities/
│   ├── 기업/A사.md
│   ├── 규제·기준/ESRS E1.md
│   ├── 산업/철강.md
│   └── 이슈/공급망 실사.md
├── runs/2026-07-06 A사 공급망 실사 대응.md   ← 요약 + run_id 링크 (전문은 1층)
└── _index/entities.json                      ← 검색용 파생 인덱스, 매번 재생성
```

엔티티 노트 예시:

```yaml
---
type: entity
entity_type: 규제·기준        # 기업|규제·기준|산업|이슈|지표|기관 (6종 고정)
aliases: [ESRS E1, 기후변화 공시기준, ESRS Climate]
updated: 2026-07-06
---
## 요약
## 축적된 사실          ← 사실마다 as_of 날짜 + 출처 run 링크 필수
## 관계                 ← 닫힌 술어 12종만: 적용된다·요구한다·공시한다·영향준다·
                          다룬다·측정한다·속한다·제정한다·상호운용된다·대체한다·
                          공급한다·관련된다 — [[위키링크]]로 표현
## 검토 필요            ← 신규 조사가 기존 지식과 다르다고 보고한 항목 append
```

- 자동 축적: 종합 단계 후 진행자 LLM에 엔티티 추출 `generate_json` 1회
  (confidence 코드 클램프 — LLM 산술 불신 원칙, 실패 시 채점처럼 삼키고 계속)
- **시드 온톨로지 약 30노트**를 repo에 포함(ESRS·CSRD·CSDDD·CBAM·K-ESG·ISSB·
  TCFD·GRI 등) — 콜드스타트·명명 파편화 완화
- 볼트 전달: MVP는 zip 내보내기/가져오기(기존 BytesIO+download_button 패턴),
  자리 잡으면 사용자별 비공개 GitHub repo + Obsidian Git 플러그인 자동 pull로 승격
- 파일명 정규화 규칙 필요: 한글+공백 파일명의 zip 인코딩(cp437/utf-8)·
  Windows 예약문자 — PowerShell 한글 이슈 전례 있음 (→ [[11 원커맨드 배포]])

### 지식 주입 — "다음 조사가 더 깊어지는" 장치

파이프라인 무수정으로 구현된다. 조사 시작 시:

1. topic+keywords를 `_index/entities.json`의 name+aliases에 매칭
2. 매칭 노트의 `## 관계` 위키링크 1홉 확장 (노트당 8,000자·총 5개 상한)
3. `brief.reference_texts["[축적 지식] 공급망 실사"]`로 주입 →
   기존 `_brief_block`이 전 단계 프롬프트에 자동 전파
4. UI: 자동 탐색 레퍼런스와 같은 체크박스 패턴으로 "축적 지식 N건 발견" 표시
5. instructions에 자동 append: 심화·업데이트 우선 + **"기존 지식과 최신 웹 결과가
   다르면 '기존 지식과의 차이' 항목으로 명시하라"** — 이 출력이 엔티티 노트의
   '검토 필요' 절로 들어가 자가 교정 루프가 됨 (판정은 사람이 Obsidian에서)

## 설계 원칙 (안전장치 — 심사에서 나온 필수 보정)

1. **환각 세탁 방지**: 주입 라벨은 "검증된 내부 지식"이 아니라
   **"과거 조사 결과(자동 생성·미검증) — 전제가 아닌 대조 대상"**. 모든 사실에
   as_of 날짜. 과거 LLM 산출물이 재인용되며 근거처럼 굳는 순환을 차단
2. **채점 단계에는 주입 금지**: 진행자가 기존 지식과 일치하는 연구원을
   높게 평가하면, 가장 가치 있는 신규 발견(기존 지식과 어긋나는 것)이
   체계적으로 저평가됨. `_brief_block` 호출부 플래그 하나로 분리 —
   이것이 유일하게 필요한 pipeline.py 수정
3. **고객사 기밀 격리**: 첨부 파일(NDA 자료) 유래 사실에 confidential 플래그,
   해당 고객사 조사에만 주입. A사 내부 자료가 B사 보고서에 섞이면 계약 위반 —
   기능이 아니라 직업적 안전장치
4. **저장 실패 폴백**: 저장 실패 시 UI 경고 + 재시도 버튼 + run.json zip 다운로드.
   20분짜리 멀티 LLM 실행이 침묵 속에 증발하는 경로를 막는다
5. 토큰 비용 주의: 주입 8k자는 (프로바이더 수 × (1+rounds) + 2)회 반복 포함됨 —
   백로그의 비용 대시보드(usage 수집)를 병행 검토

## 2단계 구현 상세 (2026-07-11)

- `core/store.py` (Streamlit 비의존) — PostgREST REST를 requests로 직접 호출
  (supabase-py 의존성 없음). `save_run`(run_id 업서트 — 재시도 안전) /
  `list_runs` / `load_run` / `record_to_state`(레코드→brief·params·PipelineResult
  복원, 필드 명시 선별로 schema 진화에 안전)
- 접근 모델: **service_role 키만** (`SUPABASE_URL`+`SUPABASE_SERVICE_ROLE_KEY`,
  서버측 Secrets/.env 전용). `ra_runs`는 RLS로 anon/authenticated 전면 차단 —
  고객사 자료가 브라우저 공개 anon 키로 열리면 안 되기 때문
- `app.py`: 성공 직후 자동 저장(`_saved_run_id` 플래그로 rerun 중복 방지),
  실패 시 결과 표시는 계속 + 경고·재시도 버튼·볼트 zip 폴백 안내(안전장치 4).
  사이드바 "🗂 이전 조사"(60초 캐시) → 불러오기 시 기존 5탭 UI로 복원되고
  볼트 zip 내보내기도 그대로 동작(run_id는 zip·아카이브 공유)
- 키 미설정 시: 저장 훅 무동작, 사이드바에 설정 안내만 표시 (기능 저하 없음)

### 활성화 절차 (사용자 1회 수행)

1. Supabase 대시보드(`unjhdoulorbmpepnkcpr`) → SQL Editor →

✅ 로컬 활성화·라이브 검증 완료 (2026-07-11): SQL 실행 + .env 키 입력 후

구현 시 발견: 사이드바가 스크립트 상단에서 먼저 렌더링되므로 저장 성공 직후
`st.rerun()`으로 목록을 즉시 갱신한다 (토스트 대신 지속 캡션 사용).
